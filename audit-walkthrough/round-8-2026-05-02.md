### C — Trust Receipt

Sibling to PrivacyStatus. `PrivacyStatus` answers "what features
COULD send data, and are they on?". `TrustReceipt` answers "show me,
right now, every storage write and every outbound request the app
made since I flipped this on."

Three capture layers:
- **Storage**: every `setJSON`/`getJSON` call publishes a set/get
  event with key + bytes (no payload).
- **Network**: `installFetchWrap()` patches `window.fetch` once at
  startup. Every request → URL + method + status + ms.
- **Cleartext-in-memory**: `recordCleartextRead(source, summary)`
  is voluntary annotation for sync-passphrase / AI-consent surfaces
  to announce they're holding sensitive data briefly.

The redaction guarantee is asserted as a unit test: PII tokens
pushed through `recordStorageEvent` and `recordCleartextRead` never
appear in the rendered panel. If a future caller wires journal text
into a trust event, the test fails with the specific token that
leaked, pointing directly at the offending call site.

The Playwright redaction spec from the wedged session was removed
in favor of this unit test (faster, stable, same property under
test). Its 332 KB of Windows-specific PNG snapshots was removed too.

### F — real-data resilience + a real bug

1000 deterministic-random user states. Caught a real DST/timezone
bug at iter=213 / seed=0x50882b05: `computeStreak (52) >
computeLongestStreak (51)`. Root cause: `src/lib/calc.ts` mixed
local-time iteration (`setDate`/`getDate`) with UTC date keys. Across
DST transitions the local-time +1 day was 25 hours, drifting one UTC
midnight.

Fix: switched all four date-walk functions
(`computeStreak`, `computeLongestStreak`, `daysSinceLastDrink`,
`computeTotalAFDays`) to use `setUTCDate`/`getUTCDate` consistently
with the existing UTC date keys. Existing 16 streak tests still
pass.

The resilience suite runs in ~12s. Failures print the
iter+seed so they're reproducible by hardcoding into a single-shot
test.
