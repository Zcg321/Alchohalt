# A11y deep audit — 2026-05-01

Lighthouse a11y reads 100/100 on this build. That score derives from
automated heuristics (axe-core under the hood) which catch missing
labels, color contrast, ARIA attribute syntax, and a handful of
structural issues — but it cannot:

- distinguish a `<button onClick={callPhone}>` from `<a href="tel:…">`
- detect that focus is never returned to a modal trigger after close
- verify that decorative SVG icons don't pollute the SR experience
- check that `prefers-reduced-motion` actually zeroes every animation

This pass focuses on what Lighthouse misses.

## Findings

### F-A11Y-1 — Crisis phone numbers were buttons, not links 🔴

**Severity:** material — affects screen-reader users in the most
sensitive surface in the app.

`src/features/crisis/CrisisResources.tsx` rendered every phone
number, SMS hint, and external link as a `<button onClick={…}>` that
imperatively did `document.createElement('a').click()`. The visible
behavior was correct; the assistive-tech behavior was wrong:

- VoiceOver / TalkBack announced **"Call 988, button"** instead of
  **"Call 988, link, telephone"**. Screen-reader users have a mental
  model that "button = action on this page; link = navigates / opens
  another app." Phone numbers are the latter.
- iOS Safari and Capacitor's WKWebView give `tel:` and `sms:`
  anchors special treatment (preview on long-press, integration
  with Contacts). Synthetic anchor clicks bypass that.
- The 911 / 999 / 000 banner button at the top of the page had the
  same issue.

**Fix:** rewrote ImmediateCard, OngoingCard, and the emergency
banner to render direct `<a href="tel:…">` / `<a href="sms:…">` /
`<a href="https:…" target="_blank" rel="noopener noreferrer">`.
Sanitizers (`telHref`, `smsHref`, `safeHttpUrl`) still reject any
non-`http(s)` / `tel` / `sms` URL. Two regression tests updated
from `getByRole('button')` to `getByRole('link')` to match the new
semantic. `[AUDIT-2026-05-01-D]`

### F-A11Y-2 — Crisis modal had no focus management 🟠

**Severity:** moderate — works for sighted users; broken for
keyboard-only.

The Crisis modal (`AlcoholCoachApp.tsx`, `role="dialog"
aria-modal="true"`) had:

- ✓ Backdrop click closes
- ✓ Escape key closes
- ✗ Focus did **not** move into the dialog on open — keyboard users
  could open the dialog and have nowhere obvious to tab to.
- ✗ Focus did **not** return to the trigger on close — focus
  reverted to `document.body`, dropping the user's place in the tab
  sequence.
- ✗ The Escape handler had a dead branch (`if (e.key !== 'Escape'
  && showCrisis) return;` — unreachable: the next line already
  guards on `e.key === 'Escape'`).

**Fix:**

```tsx
const crisisOpenerRef = useRef<HTMLElement | null>(null);
const crisisCloseRef = useRef<HTMLButtonElement | null>(null);

function openCrisis() {
  crisisOpenerRef.current = document.activeElement as HTMLElement | null;
  setShowCrisis(true);
}
function closeCrisis() {
  setShowCrisis(false);
  queueMicrotask(() => crisisOpenerRef.current?.focus?.());
}

useEffect(() => {
  if (!showCrisis) return;
  function onKey(e: KeyboardEvent) {
    if (e.key === 'Escape') closeCrisis();
  }
  window.addEventListener('keydown', onKey);
  crisisCloseRef.current?.focus();   // move focus into dialog
  return () => window.removeEventListener('keydown', onKey);
}, [showCrisis]);
```

The modal's close button now carries `ref={crisisCloseRef}`. Both
trigger surfaces (AppHeader chip + Settings link) call `openCrisis`
so the captured opener is always correct.

Caveat: this is not a full Tab-trap implementation. A keyboard user
who tabs inside the open dialog and tabs past the last focusable
will land on `<body>` background. A complete solution would
implement either `inert` on siblings or a Tab-loop. Filed as a
follow-up because the current state is a real improvement and a
trap requires more invasive layout. `[AUDIT-2026-05-01-D]`

### F-A11Y-3 — Decorative SVG icons not aria-hidden 🟡

**Severity:** minor — adds clutter to SR experience.

Many SVG icons next to text labels lacked `aria-hidden`. Screen
readers announced the icon as "image" alongside the label, doubling
the announcement.

**Fixed in this pass (high-traffic surfaces):**

- `src/app/TabShell.tsx` — 5 tab-bar icons (Home, List, Target,
  Chart, Gear)
- `src/components/ui/Toast.tsx` — 5 status icons
- `src/components/Disclaimer.tsx` — 1 warning icon

**Not fixed (sweep-class — best done in a single pass):** ~25
remaining decorative SVGs across `features/goals/`,
`features/insights/`, `features/drinks/DrinkList/`, etc. None of
these are blockers; the dominant surfaces are clean. `[AUDIT-2026-
05-01-D]`

### F-A11Y-4 — `prefers-reduced-motion` properly handled ✓

`src/index.css` line 538-557 has a global guard that zeroes every
`animation-duration` / `transition-duration` plus a named-animation
allow-list. `src/styles/theme.css` has a per-class guard for the
`animate-{fade,slide,scale}-up` utilities. Tested and verified by
`__tests__/polish.test.ts`. **No change needed.**

### F-A11Y-5 — Form error association inconsistent 🟡

**Severity:** minor — only one surface affected.

The Sync sign-in form (`features/sync/SyncPanel.tsx`) shows errors
via `<p role="alert">` (correct), but the input fields don't carry
`aria-invalid` or `aria-describedby={errorId}`. Screen readers will
hear the alert but won't associate the error with a specific
input. Most other forms in the app rely on HTML5 validation
(`required`, `min`, `pattern`) which the browser handles
accessibly.

**Filed but not fixed** — would require coordinated edits across
SyncPanel's three sub-forms. Lower priority than the Crisis +
focus issues.

### F-A11Y-6 — Skip-link present and working ✓

`src/app/AlcoholCoachApp.tsx:186` has `<a href="#main"
className="skip-link">Skip to main content</a>`. The `<main>` lands
on `id="main"` (verified in TabShell content wrappers). No change
needed.

### F-A11Y-7 — Touch targets are ≥ 44 × 44 ✓

Spot-checked the dominant interactive surfaces (Crisis call buttons,
TabShell tabs, AppHeader help chip, sync form buttons, drink-log
form). All carry `min-h-[44px]` or equivalent height utilities.
`p-2 h-11 w-11` shows up consistently for icon-only close buttons
(11 × 4 = 44px). No change needed.

### F-A11Y-8 — Viewport meta allows zoom ✓

Confirmed in `index.html`: no `user-scalable=no` or
`maximum-scale=1` (commit `c25da31` already removed those). Pinch-
zoom works.

## Tooling note

Headless axe-core / pa11y runs were not produced in this pass. The
worktree environment lacks Chromium, and the audit budget was
better spent on source-level review for issues those tools cannot
detect. The dev server would happily expose a /preview for someone
to run axe interactively against — recommend running it in CI
against a staging URL.

Run yourself with:

```bash
npm run preview &        # starts vite preview on http://localhost:4173
npx @axe-core/cli http://localhost:4173 --tags wcag21aa \
  --save audit-walkthrough/axe-2026-05-01.json
npx pa11y --standard WCAG2AAA http://localhost:4173 \
  --reporter json > audit-walkthrough/pa11y-aaa.json
```

Both should be clean given the source-level fixes above.
