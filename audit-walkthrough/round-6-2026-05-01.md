## Section A — Round-5 carry-over

### A1 — Strict TypeScript landing (`679b202`)

`tsconfig.json` had `strict: true` but `noImplicitAny: false` explicitly
turned off. With both flags on the surface looked strict but every
implicit-any callback slipped through silently. Flipped
`noImplicitAny: true`.

Damage report: 15 errors, all in two places.
- `src/store/db.ts`: 11 callbacks inside the existing
  `/* eslint-disable no-explicit-any */` helpers. Annotated each one
  (`Entry`, `Trashed` alias, `AdvancedGoal`, `DrinkPreset`,
  `HealthMetric`). The helpers themselves still take `set/get` as `any`
  because that's the zustand middleware shape, but the inner array
  callbacks now have the right type.
- `src/features/recommendations/__tests__/GoalRecommendations.smoke.test.tsx`:
  empty mock array typed as `never[]`.

All other strict-mode flags (`strictNullChecks`, `strictFunctionTypes`,
`strictBindCallApply`, `strictPropertyInitialization`, `noImplicitThis`,
`alwaysStrict`) were already on via `strict: true` and produced zero
errors.

### A3 — Hooks exhaustive-deps as error (`8e47bb0`)

`eslint-plugin-react-hooks` was missing from `devDependencies` entirely.
No `exhaustive-deps` enforcement at all. Installed; added
`plugin:react-hooks/recommended` plus explicit `rules-of-hooks` and
`exhaustive-deps` at error level.

Three real violations (not noise) fell out and got fixed:

- **`InsightsPanel.tsx`**: `goals` prop was in the `useMemo` deps but
  never read in the body. Dropped the prop entirely (call site in
  `InsightsTab.tsx` was passing `goals` it didn't need either).
- **`OnboardingFlow.tsx`**: `complete()` was a function declaration, so
  the Esc-keydown effect's deps changed every render → handler
  rebound on every keystroke. Wrapped in `useCallback` over
  `[setSettings, step]`.
- **`PremiumWellnessDashboard.tsx`**: `healthMetrics` was computed
  conditionally inside the render body (`FEATURE_FLAGS.X ? db.x : []`)
  → new array identity every render → the entire `wellnessMetrics`
  `useMemo` recomputed every render even when `db.healthMetrics` was
  unchanged. Wrapped in its own `useMemo` keyed on `db.healthMetrics`.

All three were performance bugs hiding in plain sight.

### A4 — AI-recommendations audit (`e817ddb`)

Walked `src/lib/ai-recommendations.ts` against the three audit
questions:

(a) **Useful advice — YES.** Four concrete recommendation types
    (drink-free days, weekly limit, craving awareness, monthly budget),
    each derived from real entries in the user's last 7-30 day window.
    Difficulty bucketing across readiness range works. Confidence-sorted.
    Won't recommend going backward from existing goals.

(b) **Avoids medical claims — MOSTLY.** One borderline phrase: "Notice
    the cravings. Logging them often makes them lighter" overpromises
    behavioral effect of journaling. Softened to neutral framing
    ("the log itself shows when they cluster — useful context for
    whatever you decide to try"). Regression test added so re-introducing
    the over-claim fails CI.

(c) **Real signals (not random) — YES.** Every recommendation pulls
    from `entries` field data: `stdDrinks` for weekly avg, `craving`
    for craving score, `cost` for budget projection, ts-based bucketing
    for drink-free counts. No string templates with random fillers.

Added 18 deep tests on top of the existing 3 smoke tests.

**Decision on `ENABLE_AI_RECOMMENDATIONS` flag flip**: do NOT flip in
this PR. Owner-blocking item. Logic is sound but UX framing /
positioning of "the app gives you goal recommendations" is a product
call. Recommend a follow-up PR after the owner walks the surface on a
real device, evaluates each rec-card variant in screenshots, and
decides on positioning copy.

### A5 — Streak schema migration (`2cfdf57`)

Audited the streak split. Finding: there is NO persisted streak field
in `DB`. All streak math is derived from `entries` on the fly via
`computeStreak` / `computeLongestStreak` / `computeTotalAFDays`. So
the migration concern is the data flow integrity: a future schema bump
must not silently change a user's computed longest streak.

Wrote `streak-migration.test.ts` (8 tests):
- Day-0 fresh, Day-7 mid, Day-30 power user, recovery scenario, current-
  version no-op, no-data-loss, undefined-input handling
- The owner-locked invariant: `computeLongestStreak` AFTER migration
  must be `>=` `computeLongestStreak` BEFORE migration. A migration
  that silently shrinks any user's celebrated longest streak is
  forbidden.

## Section E — Honesty pass (`d200804`)

The big finding. The About page said:

> "Your data stays on your device, encrypted with a key only you
> control. We cryptographically cannot read it."

But the local-at-rest data path goes through
`Capacitor.Preferences` with NO encryption layer. The
`SecureStorage` class existed in `src/lib/encryption/SecureStorage.ts`
(305 lines, AES-256-GCM, PBKDF2) but was **never imported anywhere**.
Pure dead code making a misleading promise.

### Findings (kept; documented)

- **`src/features/analytics/analytics.ts`** — verified no-op shim.
  All `track*` methods are noop, `captureError` logs to console only,
  no fetch calls. Honest. Regression test added.
- **`src/lib/errorReporter.ts`** — verified default reporter is
  console-only, no network call. Honest. Regression test added.
- **Cloud-backup encryption** (`src/lib/sync/`, libsodium) — wired
  and real; envelope/keys/sodium tests pass. The claim "sealed
  end-to-end with a key only your device knows" is accurate when
  sync is opted in.
- **Pricing** ($4.99 / $24.99 / $69) — these are owner-roadmap
  labels in `PLANS`, not live IAP prices. `ENABLE_SUBSCRIPTIONS`
  defaults to false, so the SubscriptionManager UI is hidden from
  users in the default build. No false price shown.
  - Owner-blocking: configure App Store Connect + Google Play IAP
    products under `com.alchohalt.app.premium_{monthly,yearly,lifetime}`
    before flipping the gate.
- **"Built by one person"** — softened to "Built by one person with
  AI assistance." More transparent; doesn't invite a journalist's
  follow-up question.

### Code changes

- `src/lib/encryption/SecureStorage.ts` — **DELETED**. Class was
  never imported anywhere. Removing dead code that backed a false
  claim.
- `src/config/features.ts` — drop `ENABLE_LOCAL_ENCRYPTION` flag
  (only the deleted `SecureStorage` referenced it).
- `src/features/settings/About.tsx` + `locales/{en,es}.json` —
  rewrite `privacy.onDevice` to match the precise / true onboarding
  language: entries live in local storage; nothing leaves the device
  on its own; cloud backup (opt-in) is the only path with E2E
  encryption; AI features (opt-in, off by default) are the only
  data-flow change.

### Regression tests (`src/__tests__/honesty-claims.test.ts`, 7 tests)

- `en.json` + `es.json` `privacy.onDevice` does NOT use the
  over-claim language ("cryptographically cannot read", "encrypted

- `en.json` `privacy.onDevice` DOES name the right scope (local
